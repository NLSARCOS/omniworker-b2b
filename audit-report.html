Fixes P0 — Templates de Codigo

P0 C1 Token Balance Race Condition
SaaS src/app/api/v1/chat/completions/route.ts:175,329 3-4 horas

Problema: Balance leido via findUnique (stale read), luego decrementado. Dos requests concurrentes pueden ambos pasar el check y deducir, llevando balance a negativo.

Codigo actual:

// Line 175: stale read
if (user.tokenBalance <= 0) {
  return NextResponse.json({ error: "Saldo insuficiente" }, { status: 402 });
}

// Line 329: atomic decrement but no guard
await prisma.license.update({
  where: { id: auth.user.licenseId },
  data: { tokenBalance: { decrement: finalCost } },
});

Fix recomendado:

// Atomic conditional update — deducts ONLY if balance sufficient
const result = await prisma.license.updateMany({
  where: {
    id: auth.user.licenseId,
    tokenBalance: { gte: finalCost }
  },
  data: { tokenBalance: { decrement: finalCost } },
});

if (result.count === 0) {
  return NextResponse.json(
    { error: "Saldo de tokens insuficiente." },
    { status: 402 }
  );
}
P0 C4-CORS CORS Reflects Any Origin
SaaS src/middleware.ts:34-41 1-2 horas

Problema: const origin = request.headers.get("origin") || "*" refleja cualquier Origin + Access-Control-Allow-Credentials: true. Cualquier sitio puede hacer requests autenticados.

Fix recomendado:

const ALLOWED_ORIGINS = new Set([
  "https://app.omniworker.com",
  "https://flux.simplex.lat",
  process.env.NODE_ENV === "development" ? "http://localhost:3000" : "",
].filter(Boolean));

const origin = request.headers.get("origin") || "";
if (ALLOWED_ORIGINS.has(origin)) {
  headers.set("Access-Control-Allow-Origin", origin);
  headers.set("Access-Control-Allow-Credentials", "true");
} else {
  headers.set("Access-Control-Allow-Origin", "");
  // Omit credentials header — block cross-origin
}

Problema: Cada refresh crea un token con familia NUEVA en vez de continuar la cadena. revokeTokenFamily() existe pero nunca se llama. Reuso de token revocado falla silenciosamente sin revocar la familia.

Fix recomendado:

// 1. Modify createRefreshToken to accept existing family
export async function createRefreshToken(

  existingFamily?: string

  const family = existingFamily || generateTokenFamily();

// 2. In refreshAccessToken, pass the family through
const newRefreshToken = await createRefreshToken(
  user.id,
  validated.family  // continue the chain

// 3. In validateRefreshToken, detect replay attacks
if (dbToken.revoked) {

P0 B3 Auxiliary Model Context Not Validated
Agent agent/context_compressor.py:913 3 horas

Problema: Contenido a comprimir enviado al modelo auxiliar sin verificar que cabe en su context window. Si main=1M y auxiliary=128K, y hay 300K+ tokens, la llamada falla y el fallback dropea turns sin resumen.

Fix recomendado:

# Before calling call_llm for summary:
aux_context = get_model_context_length(auxiliary_model)
prompt_estimate = estimate_tokens_rough(content_to_summarize + system_instructions)

if prompt_estimate > aux_context * 0.8:
    # Truncate content_to_summarize to fit
    max_chars = int(aux_context * 0.8 * 4)  # rough char estimate
    content_to_summarize = content_to_summarize[:max_chars]
    logger.warning(
        f"Compression prompt truncated: {prompt_estimate} tokens "
        f"exceeds auxiliary context {aux_context}"
    )
P0 B4 Summary Failure = Silent Data Loss
Agent agent/context_compressor.py:1648 3 horas

Problema: Default abort_on_summary_failure=False. Cuando el modelo de resumen falla, los turns del medio se dropean permanentemente y se reemplazan con un placeholder generico. Sin recovery.

Fix recomendado:

# 1. Change default
abort_on_summary_failure: bool = True  # was False

# 2. Add recovery persistence before dropping
if not summary and not self.abort_on_summary_failure:
    recovery_path = Path.home() / ".omniworker" / "recovery"
    recovery_path.mkdir(exist_ok=True)
    recovery_file = recovery_path / f"dropped_{session_id}_{int(time.time())}.json"
    recovery_file.write_text(json.dumps(
        [messages[i] for i in range(compress_start, compress_end)],
        default=str
    ))
    logger.error(
        f"Summary failed. {compress_end - compress_start} messages "
        f"saved to {recovery_file} before dropping."
    )
P0 C2-STREAM Streaming Token Billing Without Reconciliation
SaaS src/app/api/v1/chat/completions/route.ts:289,353 6-8 horas

Problema: Tokens deducidos con estimate (completionTokensEst=500) ANTES de enviar stream al client. No hay reconciliacion post-stream. Client disconnect = cobro completo.

Fix recomendado:

// Use TransformStream to count actual tokens
let actualChunks = 0;
const counter = new TransformStream({
  transform(chunk, controller) {
    actualChunks++;
    controller.enqueue(chunk);
  },
  async flush() {
    // Reconcile: refund or charge difference
    const actualCompletionTokens = actualChunks * 4; // rough
    const diff = estimatedCost - (promptTokensEst + actualCompletionTokens);
    if (diff > 0) {
      await prisma.license.update({
        where: { id: licenseId },
        data: { tokenBalance: { increment: diff } }
      });
    }
  }
});

return new Response(
  aiResponse.body.pipeThrough(counter),
  { headers: streamHeaders }
);
